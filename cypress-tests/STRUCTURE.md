# Project structure — file by file

Every file that is committed has a deliberate reason to exist and a deliberate location. This document explains both.

---

## Repo root

### `package.json`
Single source of truth for dependencies and npm scripts. Must live at root — npm resolves the dependency graph and lock file from the directory containing `package.json`. Moving it anywhere else breaks `npm install` and all workspace tooling.

> **`npm install`** reads `package.json`, resolves all dependencies, and creates `node_modules/`. It must be run from the directory containing `package.json`.

### `package-lock.json`
Auto-generated by npm. Locks the exact version of every transitive dependency so any `npm install` install — locally, in CI, on a reviewer's machine — produces identical `node_modules`. Committed intentionally; updating it without cause would invalidate reproducibility guarantees.

### `.gitignore`
Git configuration — must live at repo root. Excludes generated output (`cypress-tests/reports/`, `cypress-tests/.nyc_output/`, `app-instrumented/`), editor noise (`.DS_Store`), secrets (`*.local.md`), and `node_modules/`.

### `.nycrc.json`
Configuration for **nyc** (Istanbul CLI), which instruments `app/js` and generates coverage reports. Lives at root because nyc resolves all paths relative to process working directory, which is always the repo root when invoked via npm scripts. Placing it inside `cypress-tests/` caused a path resolution split: `nyc report` (root cwd) and `@cypress/code-coverage/task` (Cypress cwd) resolved `temp-dir` differently, producing a nested `cypress-tests/cypress-tests/` artefact folder. Root placement is the single source of truth for both consumers.

Key decisions:
- `include: ["app/js/**/*.js"]` — measures only app source code, never test helpers
- `reporter: ["text", "text-summary"]` — terminal-only output, no HTML files to commit or clean up
- `temp-dir: "cypress-tests/.nyc_output"` — keeps coverage data collocated with the test project
- `all: true` — reports 0% for files never imported by any test, preventing false 100% totals

### `README.md`
Entry point for anyone who clones the repo. Covers quick start, layout, design decisions, scripts, CI, and known coupling issues. Lives at root because GitHub, GitLab, and most tooling render `README.md` from the repo root automatically.

### `.github/workflows/ci.yml`
GitHub Actions pipeline. Must live in `.github/workflows/` — that is the path GitHub scans for workflow definitions. Runs lint + Cypress on Chrome and Firefox in parallel on every push to `main` and on every pull request.

## `app/`

The provided SPA under test. Not written by the automation engineer — treated as a black box. Kept untouched except for the coverage instrumentation step (`nyc instrument app/js app-instrumented/js`) which writes output to a separate `app-instrumented/` directory and never modifies `app/` itself.

- `app/index.html` — entry point, loads `js/index.js` and `js/users.js`
- `app/js/index.js` — application logic (login, logout, navigation, session)
- `app/js/users.js` — hardcoded user credentials, the only "database"
- `app/css/style.css` — stylesheet
- `app/img/` — background images and icon

## `cypress-tests/`

Everything related to the automation framework lives here. Keeping it in a dedicated folder with `cypress.config.js` at its root lets Cypress be invoked with `--project cypress-tests` from the repo root — clear separation between the system under test (`app/`) and the framework that tests it.

The folder is named `cypress-tests` rather than just `tests` to make the technology explicit at a glance and to avoid the common confusion of a generic `tests/` folder that could contain unit, integration, or other test types.

### `cypress-tests/cypress.config.js`
Cypress project configuration. Must live at the root of the Cypress project (i.e. `cypress-tests/`) so `--project cypress-tests` can locate it. Contains only non-default values — no commented-out defaults, no placeholder keys. Notable decisions:
- `allowCypressEnv: false` — disables `Cypress.env()` which leaks env vars to browser scope; deprecated in Cypress 15
- `retries: { runMode: 1, openMode: 0 }` — one retry in headless CI only; retries in interactive mode hide flakiness during development
- `reporter: require.resolve(...)` — absolute resolution avoids the `MODULE_NOT_FOUND` error that occurs when Cypress looks for the reporter relative to its own binary location rather than the project root
- `autoOpen: true` — HTML report opens in browser automatically after a headless run

### `cypress-tests/.eslintrc.json`
ESLint config scoped to this folder only. Lives in `cypress-tests/` so it does not affect the `app/` source. Contains the `cypress` plugin globals (`cy`, `Cypress`, `describe`, `it`, etc.) that would otherwise produce "undefined variable" lint errors.

### `cypress-tests/.prettierrc`
Prettier formatting config scoped to test files. Single quotes, 2-space indent, trailing commas. Lives in `cypress-tests/` — same reasoning as `.eslintrc.json`.

### `cypress-tests/FINDINGS.md`
QE deliverable: nine bugs found during black-box testing (including one critical security issue), coverage gaps, and broader recommendations. Lives in `cypress-tests/` because it is a product of the test activity, not of the application itself. It is committed so it travels with the code and is visible on GitHub.

## `cypress-tests/tests/`

Contains only `*.spec.js` files — nothing else. Named `tests/` rather than `e2e/` because the outer folder already establishes the Cypress context; inside it, "tests" is the clearest name for where the test cases live.

Each spec is named after the feature it covers, not after a page or component. This makes it immediately obvious what breaks when a spec fails.

| File | What it covers |
|---|---|
| `login-valid.spec.js` | Data-driven happy-path login for all valid users; session persistence across page reloads |
| `login-invalid.spec.js` | 12 negative cases: wrong credentials, empty fields, case sensitivity, SQL injection, two XSS payloads, unicode, whitespace, 500-char strings |
| `logout.spec.js` | Dropdown toggle (open/close), logout clears session and returns to login |
| `menu.spec.js` | Navigation bar items exist and respond to clicks |
| `accessibility.spec.js` | axe-core audit on login, home, and open dropdown; full keyboard-only authentication flow |

## `cypress-tests/fixtures/`

### `cypress-tests/fixtures/loginData.js`
Test data separated from test logic. Exports `defaultUser`, `validInputs`, and `invalidInputs`. Specs import from here — changing a test case means editing the fixture, not the spec. The invalid inputs dataset deliberately includes security payloads (SQL injection, XSS) to verify the app handles them without crashing or executing.

Lives in `fixtures/` rather than inline in specs to keep specs readable and to make the data reusable across multiple specs without duplication.

## `cypress-tests/support/`

Shared helpers loaded by Cypress before every spec. Nothing in `support/` contains test assertions about specific features — it is infrastructure only.

### `cypress-tests/support/index.js`
Cypress support file — the single entry point Cypress loads before each spec (configured via `supportFile`). Registers third-party plugins (`cypress-axe`, `cypress-real-events`, `cypress-mochawesome-reporter`) and sets up three global hooks:
- `before` — resets the nyc coverage counter at the start of a run
- `afterEach` — merges `window.__coverage__` into the nyc temp file after each test
- `after` — generates the coverage report once all tests in the file complete

### `cypress-tests/support/session.js`
Encapsulates `cy.session()` so specs never repeat login steps. `loginAs(user)` creates a named session on first call; Cypress restores it from cache on subsequent calls in the same run. The `validate` callback re-checks `localStorage` on restore — if the session is invalid it re-runs setup rather than silently continuing with a broken state. Exported as standalone functions (not a class) because there is no state to manage.

### `cypress-tests/support/a11y.js`
Single wrapper around `cy.injectAxe()` + `cy.checkA11y()`. The `includedImpacts: ['serious', 'critical']` filter is centralised here so the threshold is changed in one place if requirements change. Named `a11y.js` — the standard abbreviation for "accessibility" in the frontend world.

### `cypress-tests/support/pages/`
Page Object Model classes. Each file represents one page or major UI region. Specs interact with the application exclusively through these objects — selectors are never written in spec files directly.

**`LoginPage.js`**
Actions: `typeEmail`, `typePassword`, `submitLogin`, `login`, `submitWithEnter`.
Assertions: `verifyLoginFormVisible`, `verifyEmailEmpty`, `verifyPasswordEmpty`, `verifyEmailFocused`.
Exported as a singleton `onLoginPage` so specs read as prose: `onLoginPage.login(user)`.

**`HomePage.js`**
Actions: `openUserMenu`, `closeUserMenu`, `logout`, `clickNav`, keyboard helpers.
Assertions: `verifyNavVisible`, `verifyAvatarVisible`, `verifyUserMenuOpen`, `verifyUserMenuClosed`, etc.
Toggle methods (`openUserMenu`, `closeUserMenu`) are intentionally non-idempotent — they assert the pre-condition before acting. A method that silently skips when the menu is already open would mask bugs in the toggle behaviour itself.
Exported as a singleton `onHomePage`.

## Generated / gitignored

| Path | What it is |
|---|---|
| `app-instrumented/` | Copy of `app/` with coverage counters injected by `nyc instrument`. Regenerated on every run. Never committed. |
| `cypress-tests/.nyc_output/` | Raw per-test coverage JSON written by `@cypress/code-coverage`. Input for `nyc report`. Never committed. |
| `cypress-tests/reports/` | mochawesome HTML report, screenshots, and videos. Never committed — uploaded as a CI artefact instead. |
| `cypress-tests/downloads/` | Files downloaded by Cypress during tests. Never committed. |
